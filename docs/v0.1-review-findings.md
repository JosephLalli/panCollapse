# v0.1 Pre-Release Review Findings

Adversarial read-only oracle review of the PathTally (D048/D049) implementation for a v0.1
release, 2026-07-02. Original verdict: FAIL as an unqualified release; PASS WITH RISKS only if
v0.1 was explicitly rescoped to small-scale with the blockers documented as known limits.

Resolution (2026-07-03/04): all three blockers (B1-B3) and every non-blocking finding
(N1-N11) are now `resolved`. B3 removed the single-chunk u32 ceiling, so the small-scale
rescoping is no longer required. Full CTest suite: 30/30 passing.

Status key: `open`, `resolved`, `wontfix`, `deferred`.

## Blocking

### B1 — No hermetic alevin-fry interoperability test in CI — `resolved`
Fixed by the `pathtally_af_*` CTest chain (`tests/vg/CMakeLists.txt`): a hand-authored graph +
GAMP go through `panCollapse convert`, then the supported alevin-fry runs
`generate-permit-list -> collate -> quant`, and `pathtally_af_verify.py` asserts the exact
cell-by-gene matrix (`AAACCCAAGTTTGGGA`: GENE1=2, GENE2=1). Independent of the RAD layout the
Python oracle assumes, so a layout regression that still parses would fail the matrix check.

Retiring the old `phase2_*`/`phase3_*` fixtures removed the only
`generate-permit-list -> collate -> quant` conformance check with no replacement
(`tests/vg/CMakeLists.txt`). The tool's primary contract (emit alevin-fry-consumable RAD) is
unverified in CI, and the Python oracle re-implements the same RAD layout so it would ratify a
layout regression. Violates `docs/product-spec.md` Section 11-12 and `docs/validation-contract.md`
Section 7. Fix: a tiny self-contained CTest that runs the supported alevin-fry over a generated
`map.rad` and asserts the expected matrix.

### B2 — Fidelity check and failure modes un-exercised in CI — `resolved`
Fixed by new CTests (`tests/vg/CMakeLists.txt`): `pathtally_fid_build` simulates reads from the
HST paths and maps them with `vg mpmap` so the GAMP carries vg's own `Subpath.score`, then
`pathtally_fid_flat` and `pathtally_fid_qualadj` assert the flat and quality-adjusted per-node
scorers reproduce those scores; `pathtally_smoke_stdin` proves `--gamp -` is byte-identical to
file input; `pathtally_badmol_skip`/`pathtally_badmol_fail` cover molecule-identity skip (with
summary-counter assertions) and the fail policy (WILL_FAIL); `pathtally_recur_fail` covers the
grouping-recurrence abort (WILL_FAIL). Reverse-strand (`_R`) HST orientation is exercised by the
`NM_2.2_R1` path in the smoke oracle chain.

Originally: `pathtally_score_test` only ran the synthetic asserts unless a GAMP path was passed on
argv, and the CTest passed none (`tests/vg/pathtally_score_test.cpp:249-257`).

### B3 — Single-chunk RAD writer with a u32 byte ceiling — `resolved`
Fixed in `RadStreamWriter` (`src/main.cpp`): records roll into a new chunk once the current chunk
would exceed a configurable byte budget (`PANCOLLAPSE_MAX_CHUNK_BYTES`, default 1 GiB), each chunk
header is seek-and-backpatched with its true `nbytes`/`nrec`, and the file header's `num_chunks` is
backpatched at finalize. Verified byte-identical to the single-chunk default and, under a forced
one-record-per-chunk split (`pathtally_smoke_*_multichunk`), still exactly matched by the oracle and
consumable by alevin-fry.

## Non-blocking

- **N1 (High) — Stale canonical spec vs D048/D049** — `resolved`. `docs/product-spec.md`
  Sections 11-12 and `docs/validation-contract.md` Section 7.8 were reconciled to the implemented
  behavior: seek-and-backpatched `num_chunks` (D049) instead of `num_chunks = 0`, and the removal
  of the `score_removed_targets` score-window counter and the mixed-orientation-drop counter
  (both superseded by D048's top-score-plus-ties and majority-of-bases orientation).
- **N2 (High) — Unbounded `completed_names`** — `resolved (documented)`. Comment at the
  `completed_names` declaration in `src/main.cpp` states the O(distinct read-group names) memory
  cost and the deliberate memory/correctness tradeoff; the streaming claim is qualified in
  `docs/conversion-algorithm.md` "Output and determinism".
- **N3 (Medium) — Per-subpath node-score summation** — `resolved (working as intended)`. See the
  analysis below; the summation is the correct and forced choice under the no-enumeration design.
  Follow-up: document the semantics explicitly and add a branching-multipath fixture.
- **N4 (Medium) — No GAMP/xg node-id-space check** — `resolved`. The `lookup` closure in
  `src/main.cpp` now throws `std::runtime_error` when a mapped node ID is absent from the graph
  (caught at the top level -> stderr message -> exit 1); a node that exists but lies on no HST path
  continues to produce empty tallies and exit 0. Regression covered by `pathtally_badnode_fail`
  (WILL_FAIL) in `tests/vg/CMakeLists.txt`.
- **N5 (Medium) — qualadj unchecked positional reads** — `resolved`. Added bounds checks in
  `src/pathtally_qualadj.hpp`: `read_start_offset` is validated against both string sizes at
  `score_subpath` entry, and before every match/mismatch inner loop the computed read and node
  end offsets are checked against `read_seq`, `read_qual`, and `node_seq` sizes; a violation
  throws `std::runtime_error` with the out-of-range endpoint and container size. Valid inputs
  produce identical scores; all 29 CTests pass.
- **N6 (Medium) — qualadj determinism is per-platform** — `resolved (documented)`.
  `std::exp/log/pow/round` matrix construction can differ by +/-1 across libm, flipping a
  winner tie and thus RAD bytes across machines. Flat is unaffected. Documented in
  `docs/input-output-contract.md` (the "Byte-identical output" bullet) and
  `docs/conversion-algorithm.md` (the "Output and determinism" section).
- **N7 (Medium) — Hardcoded machine-specific build paths** — `resolved`. `cmake/PanCollapseVg.cmake`
  now defaults both `PANCOLLAPSE_VG_ROOT` and `PANCOLLAPSE_ABSL_PKGCONFIG` to empty; an unset VG
  root triggers a clear fatal error pointing to the required `-D` flag; the Abseil pkgconfig entry
  is applied only when the variable is explicitly set and the path exists; `README.md` documents all
  configure variables in a table.
- **N8 (Low) — No `-Wall -Wextra`/`-Werror`** — `resolved`. Added via an INTERFACE library
  `panCollapse::warnings` in the root `CMakeLists.txt`; each project-owned target (`panCollapse`,
  `vg_link_smoke`, `pathtally_score_test`, `pathtally_pipeline_test`) links it PRIVATE so the flags
  apply only to project translation units and do not propagate to VG/third-party headers.
  `-Werror` deliberately omitted to avoid breakage from third-party header noise.
- **N9 (Low) — No `--version`/`--help`** — `resolved`. `PANCOLLAPSE_VERSION` surfaced via compile
  definition; `--version`/`-V` print `panCollapse 0.1.0` to stdout and exit 0; `--help`/`-h` and
  no-args print usage to stdout and exit 0; bad arguments still exit 1.
- **N10 (Low) — Partial `map.rad` on failure** — `resolved`. `RadStreamWriter` now writes to
  `<out_dir>/map.rad.tmp` and renames to `map.rad` atomically only after `finalize()` completes;
  the destructor removes the temp file on any earlier exit. Regression covered by
  `pathtally_recur_no_partial` (asserts non-zero exit AND absence of `map.rad`).
- **N11 (Low) — Summary omits per-group target-size distribution** — `resolved`. Added
  `emitted_target_count_histogram` to `summary.tsv`: semicolon-separated `target_count:group_count`
  pairs sorted ascending by target count.

## N3 analysis — per-subpath node-score summation is correct

`tally_read_group` (`src/pathtally.hpp`) scores each subpath once and adds each node's score to
the HST paths crossing it, with no per-node dedup across subpaths. The concern was double-counting a
node shared by mutually-exclusive sibling subpaths.

Reasoning:

1. PathTally deliberately does not enumerate complete traversals (D048); the subpath is the atomic
   unit of aligned evidence.
2. A multipath alignment expresses branches and merges as `next`/`connection` edges between
   single subpaths, not by duplicating nodes across subpaths. Each subpath is scored exactly once,
   so DAG merge/branch structure is not over-counted. A node appears in two subpaths only when the
   read genuinely aligns to it in two distinct places/ways.
3. To "dedup" such a node you must decide the two occurrences are the same evidence rather than a
   genuine repeat. That distinction requires reasoning about which subpaths form which traversal —
   i.e. exactly the enumeration D048 removed. Any dedup that skips enumeration silently corrupts the
   legitimate repeat case.
4. The residual over-count (read-overlapping alternative subpaths that share a node) is symmetric
   across every HST crossing that node and reflects genuine mapping multiplicity, not a ranking
   artifact against a clean baseline.

Conclusion: summation is the only aggregation consistent with the no-enumeration design; the
alternatives either reintroduce the exponential traversal enumeration or make an incorrect dedup
assumption. Working as intended. Follow-up: state the per-subpath-occurrence semantics explicitly in
`docs/conversion-algorithm.md` and add a branching-multipath fixture to lock the behavior.
